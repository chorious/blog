## 缘起

五一假期前后，花两天时间从零到一搭建了一个完整的职业锚测评系统。起因很简单：一位朋友想做一套在线问卷，用于团队内部的职业倾向评估。现成的 SaaS 工具要么太贵，要么数据不可控。索性自己动手，后端用 Go + SQLite，前端用 React + Tailwind，静态页面往 GitHub Pages 一丢，后端通过 Cloudflare Tunnel 暴露出来——零服务器成本，数据还攥在自己手里。

## 技术选型

| 层级 | 技术 | 理由 |
|---|---|---|
| 前端 | React 18 + TypeScript + Vite | 开发快，类型安全，构建体积小 |
| 样式 | Tailwind CSS | 原子化，改UI不用切文件 |
| 后端 | Go 1.24 + Gin | 单文件编译，SQLite 内嵌，部署极简 |
| 数据库 | SQLite (modernc.org/sqlite) | 纯 Go 实现，无 CGO，跨平台零依赖 |
| 部署 | GitHub Pages + Cloudflare Tunnel | 前端免费托管，后端隧道穿透 |

整个项目最爽的一点是：**后端只有一个 `main.go` 文件**。从 `initDB` 到路由注册到 HTTP 服务启动，全部写在一起。对于这种工具型项目，过度拆分反而是一种负担。

---

## 核心机制：10 题定 8 锚

系统基于 Edgar Schein 的职业锚理论，将职业驱动力分为 8 种类型：技术职能型(TF)、管理型(GM)、自主独立型(AU)、安全稳定型(SE)、创业型(EC)、服务型(SV)、挑战型(CH)、生活方式型(LS)。

每道题 8 个选项分别对应 8 种锚，用户答完 10 题后统计各选项出现次数，得分最高的即为 dominant anchor。如果两个锚同分，则显示「双锚型」。这个计分逻辑极其朴素——就是 `map[string]int` 做计数器——但效果出奇地好，因为 10 题 × 8 选项的排列组合本身就足以区分倾向。

## 两天踩坑实录

### Day 1：从 MVP 到可交付

**上午：数据流贯通**

最大的改动是「必填联系信息」。原始 fork 只有昵称，提交时不存任何联系方式，这对于实际收集数据来说完全不可用。加了姓名和手机号两个字段，前后端都做校验：

- 前端：正则 `^[一-龥]+$` 限中文，`^1[3-9]\d{9}$` 限 11 位手机号
- 后端：同样的正则再验一遍，不信任前端
- 数据库：`ALTER TABLE ADD COLUMN` 做兼容迁移，老数据默认值空字符串

**下午：UI 换肤**

原始模板的紫粉渐变 + 毛玻璃效果，放在「职业测评」这个语境下显得过于娱乐化。改成商务蓝白风：

- 主色 `blue-800`（#1e40af），实色无渐变
- 卡片纯白 + 轻阴影 + 细灰边框，去掉毛玻璃
- 按钮去弹跳、去描边发光，hover 只做底色加深

改造完第一眼就觉得「对味了」。Tailwind 的好处这时候体现出来：改色不用翻 CSS 文件，全局搜 `purple` 和 `pink` 替换掉就行。

**晚上：云部署版**

顺手给有服务器的朋友准备了一个 Docker 部署分支，多阶段构建：Node 编译前端 → Go 编译后端 → Alpine runtime，最终产出单个二进制文件，内置前端 dist，数据卷挂到 `./data/`。

### Day 2：魔鬼藏在细节里

**版本同步：GitHub Pages 的 5 分钟幽灵**

这是整个项目中最纠结的一个体验问题。

我的部署流程是：前端静态文件 push 到 GitHub Pages，后端二进制在本地 Windows 运行，通过 Cloudflare Tunnel 暴露。每次发版时，先 push 前端代码（触发 Pages 构建），再重启本地后端（读取新的版本号）。

问题来了：**GitHub Pages 的部署有延迟**，有时长达 5 分钟。这意味着后端已经重启并报告新版本了，但用户刷新页面拿到的还是旧版前端。前端检测到版本不匹配 → 提示刷新 → 用户刷新 → 仍然拿到旧版 → 再次提示刷新……无限循环。

解法：**让后端等前端**。新增一个 `VERSION` 文件作为单一版本源，后端每次响应版本请求时，先去轮询 GitHub Pages 上的 `version.json`，确认前端已经部署到最新版后，才把版本号暴露出来。这样前端永远不可能「超前」于实际部署状态。

```go
// 伪代码：后端版本处理逻辑
deployedVersion := readVersionFile() // 本地 VERSION 文件
frontendVersion := pollGitHubPages()  // 轮询 Pages 上的 version.json

if compareVersion(deployedVersion, frontendVersion) <= 0 {
    return deployedVersion // 前端已同步，正常返回
} else {
    return frontendVersion // 前端落后，返回旧版，避免触发刷新
}
```

前端侧也加了防护：刷新失败后显示 10 秒倒计时灰遮罩，再弹「立即刷新」。同时修复了一个 tab 切换误报刷新的 bug——用 `PerformanceNavigationTiming.type` 区分真正的页面重载和标签页切换。

**缓存失效：CDN 不承认你的更新**

GitHub Pages 默认缓存策略很激进，用户可能几个小时都拿不到最新构建。解法是两管齐下：

1. `vite.config.ts` 里用 `define` 注入 `__APP_VERSION__`，值为构建时的 ISO 时间戳
2. `index.html` 里加 `<meta http-equiv="Cache-Control" content="no-cache">`

本质上是让每次构建的 HTML 都是「新的」，逼迫 CDN 回源。

**分享图生成：DOM 截图的坑**

结果页需要生成一张可分享的卡片。用 `html-to-image` 把隐藏的 DOM 节点转成 PNG，再用 `qrcode` 在底部附加二维码。

坑点：

1. **尺寸问题**：ShareCard 的宽度固定 375px（手机屏宽），但生成的图片在桌面端预览时显得太小。反复调了几轮，最终预览弹窗限制 `max-w-[300px]`，约 80% 实际尺寸，视觉舒适。
2. **二维码颜色**：默认黑白色太突兀，改成主题蓝 `#1e40af`，和卡片 header 的 gradient 呼应。
3. **复制到剪贴板**：iOS Safari 不支持 `ClipboardItem` 写图片，降级提示用户保存后手动分享。

**UI 精修：Design Token 与去 Emoji**

Opus 4.7 在审阅时指出三个问题：emoji 滥用、圆角混乱、次要按钮色相竞争。遂做了一次精修：

- 用 `rounded-card`（12px）、`rounded-control`（8px）、`rounded-pill`（9999px）三档 token 替换之前混用的 `rounded-lg/xl/2xl/full`
- 状态提示去掉 `⏳ ✅ ❌`，改成带柔和底色和边框的 soft chip（`bg-blue-50 + border-blue-100`）
- 按钮分三档：primary 实色 / secondary outline / text 纯文字，消灭色相竞争

**测试覆盖**

给最复杂的版本轮询逻辑写了 14 个 Go test case，覆盖正常同步、前端落后、后端落后、相等、空值、格式错误等场景。跑 `go test` 全部 PASS，心里踏实不少。

## 部署架构

```
用户浏览器
    │
    ├──→ GitHub Pages (静态前端，免费 CDN)
    │
    └──→ Cloudflare Tunnel ──→ 本地 Windows (Go + SQLite)
                               (Cloudflare 边缘网络转发)
```

这套架构的好处是**零服务器成本**，前端全球 CDN 加速，后端只有我一个人访问管理后台时需要开 Tunnel。缺点是 Tunnel URL 每次重启会变，需要手动更新前端硬编码的 API 地址。如果以后流量稳定，可以考虑换固定域名的内网穿透，或者直接上云服务器。

## 一些教训

1. **修改 Go 源码后必须重新编译**。听起来很基础，但如果你在 Windows 上跑的是 `go build` 出的 exe，改代码后忘记重新 build，会陷入「明明代码改了为什么行为没变」的困惑——我就这么浪费了 20 分钟。

2. **不要信任前端的任何计算**。目前评分逻辑还在前端做，后端只存结果。这意味着懂行的用户可以伪造答案得到任意想要的职业锚。下一步必须把计分搬到后端，前端只传原始选项。

3. **版本号管理要早做**。如果等项目复杂了再加版本同步机制，改造成本会高很多。一个 `VERSION` 文件 + 一轮轮询就解决的问题，不要等到用户投诉「为什么刷新没用」才去修。

## 总结

两天时间，从 fork 一个 MBTI 测试模板，到改造成完整的职业锚测评系统，再到解决部署同步、缓存失效、分享图生成等一系列工程细节——这个过程再次印证了「完成比完美重要」。如果一开始就想着把后端计分、输入校验、WAL 模式全部做完美，可能两天后还在写设计文档。

先让系统跑起来，让用户能用，再逐步填坑。现在的版本号停在 **v0.0.17**，还有不少事情可以做：后端独立计分、管理后台可视化、多语言支持、真正的云部署。但至少此刻，它已经是一个「能用」的系统了。

以上即是全部开发记录，供以后查阅。
